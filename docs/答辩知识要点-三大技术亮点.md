## 一、按标签 RAG（Label-wise Retrieval）

### 1.1 知识原理

**传统 RAG 的做法：**

- 用户提问 → 将整个问题与知识库做一次向量检索 → 取 top-k 最相关文档 → 拼成上下文给 LLM。
- 问题：检索是「全局」的，不区分「法律关系」「争议焦点」「风险点」等不同维度的需求，容易混入无关内容或漏掉某一类知识。

**按标签 RAG 的原理：**

- 把法律分析拆成多个维度（标签），如：法律关系、争议焦点、风险点、证据与程序注意、处理建议。
- 对每个标签，把「标签 + 案情」当作独立查询去检索，得到该维度最相关的法条。
- 最后按标签组装上下文，每个维度都有针对性知识，模型在做对应部分分析时能看到更贴切的法条。

**技术实现：**

- 检索：TF-IDF + 余弦相似度（字符级 2–4 gram，适合中文）。
- 关键：不是「一次检索全库」，而是「按标签多次检索，再拼装」。

**优点：**

- 维度清晰：每个输出项都有专门的法条支撑。
- 减少干扰：法律关系检索不会混入证据类条款。
- 适配法律分析结构，可解释性更好。

### 1.2 项目应用

**位置：** `agent/scripts/structured_pipeline.py`

**核心函数：**

```python
def retrieve_by_labels(kb, query, labels=LABELS, top_k=2):
    res = {}
    for lab in labels:
        q = lab + " " + query   # 关键：标签与案情拼接成查询
        res[lab] = retrieve(kb, q, top_k=top_k)
    return res
```

**流程：**

1. `LABELS = ["法律关系", "争议焦点", "风险点", "证据与程序注意", "处理建议"]`
2. 对每个 label，用 `label + " " + query` 做一次 TF-IDF 检索，取 top-2。
3. 用 `build_kb_text_by_labels()` 按标签组织成「法律关系：…\n争议焦点：…」的文本块。
4. 将该文本块作为「参考知识」写入 prompt，再交给 LLM 生成结构化分析。

**调用示例：**

```bash
python scripts/structured_pipeline.py --query "案情描述..." \
  --kb_path data/knowledge/legal_articles.json \
  --use_rag --rag_by_label
```

**答辩可强调：**

- 与通用 RAG 不同，我们的检索是「按分析维度」定向的。
- 适配法律场景：法律分析天然有结构，按标签检索能更精准匹配。

## 二、双路路由（Local/Cloud Routing）

### 2.1 知识原理

**问题：**

- 纯本地小模型：成本低，但复杂/偏门问题容易答不好。
- 纯云端大模型：效果好，但调用贵、有延迟。

**双路路由：**

- 先做 RAG 检索，得到相似度分数（如 max_score）。
- 若分数高于阈值：认为知识库里有足够支撑，交给本地 Qwen 处理。
- 若分数低于阈值：认为知识不够，交给云端通义千问兜底。

**核心思想：**

- 用「检索相似度」近似表示「能否可靠回答」。
- 简单题用本地，难题用云端，在效果和成本之间做自适应平衡。

**技术细节：**

- 相似度：来自 TF-IDF 的余弦相似度。
- 阈值：可调（如 0.15），按业务需求调节本地/云端比例。

### 2.2 项目应用

**位置：** `agent/scripts/model_api_router.py`

**核心逻辑：**

```python
# 1. 按标签检索，得到每条文知识的相似度分数
label_articles = sp.retrieve_by_labels(kb_data, query, sp.LABELS, top_k=2)
max_score = max(a.get("score", 0.0) for arts in label_articles.values() for a in arts)

# 2. 决策：高分走本地，低分走云端
if max_score >= threshold:
    # 本地 Qwen + RAG 知识
    text = sp.generate(tokenizer, model, prompt, ...)
    return {"source": "student", "structured": parse_structured(text)}
else:
    # 云端 DashScope 兜底
    api_text = call_api_fallback(query, kb_text)
    return {"source": "api_fallback", "provider": "dashscope", ...}
```

**调用示例：**

```bash
python scripts/model_api_router.py --query "案情..." \
  --threshold 0.15 --fallback_api dashscope
```

**参数说明：**

- `--threshold`：相似度阈值，越高越倾向走云端。
- `--fallback_api dashscope`：低于阈值时调用通义千问。

**答辩可强调：**

- 不是简单的「要么本地要么云端」，而是根据问题难度动态选择。
- 检索相似度作为路由依据，逻辑简单、可解释。

## 三、知识蒸馏与 LoRA 微调

### 3.1 知识原理

**知识蒸馏（Knowledge Distillation）：**

- 教师模型：大模型（如通义千问），能力强但笨重。
- 学生模型：小模型（如 Qwen 0.5B），轻量但能力弱。
- 做法：用教师对大量样本生成答案，把这些「输入-输出对」当作训练数据，训练学生模仿教师。
- 结果：学生在小参数量下也能学到教师的推理风格和能力。

**LoRA（Low-Rank Adaptation）：**

- 微调时不动原有权重，只加一组低秩矩阵（如 r=16 的 A、B 矩阵）。
- 训练时只更新 A、B，参数量远小于全量微调（通常只有原模型的 0.1% 量级）。
- 推理时：base 模型 + LoRA  adapter，计算量和显存占用都小。

**QLoRA：**

- 在 LoRA 基础上，对 base 模型做 4-bit 量化。
- 显存再降，可在消费级 GPU 甚至 CPU 上做微调。

### 3.2 项目应用

**整体流程：**

```
判决书/案情种子 → distill_qwen_api.py（通义生成）→ 蒸馏数据
                                              ↓
                         finetune_qwen_lora.py ← 清洗后的 JSONL
```

**1. 知识蒸馏（教师生成数据）**

**位置：** `agent/scripts/distill_qwen_api.py`

- 输入：包含案情的 JSONL（每行 `{"input": "案情..."}`）。
- 调用通义千问 `qwen2.5-instruct` 做法律分析。
- 输出：`instruction + input + output` 的 JSONL，供微调用。

```python
# 核心：云端大模型生成训练标签
resp = Generation.call(model="qwen2.5-instruct", prompt=prompt)
out.append({
    "instruction": "请对以下案情进行法律辅助判断",
    "input": case,
    "output": resp.output_text
})
```

**2. LoRA 微调**

**位置：** `agent/scripts/finetune_qwen_lora.py`

**LoRA 配置：**

```python
LoraConfig(
    r=16,                    # 低秩维度
    lora_alpha=32,           # 缩放因子
    lora_dropout=0.05,
    target_modules=["q_proj","k_proj","v_proj","o_proj","up_proj","down_proj","gate_proj"],
    task_type="CAUSAL_LM",
)
```

- 目标模块：注意力中的 Q、K、V、O 和 MLP 的 up/down/gate。
- 数据格式：`instruction + input + output`，训练时对 prompt 部分做 labels 掩码（-100），只对 output 计算 loss。

**可选 QLoRA：**

```bash
python scripts/finetune_qwen_lora.py --use_qlora
```

- 4-bit 量化 base 模型，显存需求更低。

**调用示例：**

```bash
# 1. 蒸馏：通义生成训练数据
python scripts/distill_qwen_api.py --input_jsonl seeds.jsonl --output_jsonl distilled.jsonl

# 2. 微调（需 prepared 的 JSONL）
python scripts/finetune_qwen_lora.py --dataset_path data/processed/cleaned_cases.json \
  --base_model Qwen/Qwen2.5-0.5B-Instruct --output_dir data/finetune/qwen_lora
```

**答辩可强调：**

- 蒸馏：用大模型能力弥补小模型不足，无需人工标注。
- LoRA：只训少量参数，保留 base 通用能力，专注法律领域。
- QLoRA：在显存有限时仍可微调，适合个人电脑。

## 附录：答辩常见追问速答

| 追问 | 建议回答 |
|------|----------|
| 为什么用 TF-IDF 不用向量？ | 项目优先考虑轻量、无 GPU 部署；TF-IDF 字符 n-gram 对中文效果尚可，后续可升级为向量检索。 |
| 阈值 0.15 怎么定的？ | 经验值，可据实际效果调；越高越倾向云端，越低越倾向本地。 |
| LoRA 的 r=16 有什么依据？ | 常用范围 8–64，16 是常见折中；可按显存和数据量再调。 |
| 蒸馏数据质量如何保证？ | 通义千问本身质量较高；可加人工抽检或规则过滤异常输出。 |
| 按标签检索会不会漏知识？ | 每标签取 top-2，覆盖主要维度；若担心可增大 top_k，但会增加上下文长度。 |
